================================================================================
ESTIMATIZ — NOTICE TECHNIQUE COMPLÈTE
================================================================================
Outil d'estimation immobilière basé sur les données DVF (Demandes de Valeurs
Foncières) de la Direction Générale des Finances Publiques.

Environnement local : XAMPP (Apache + MySQL) • PHP 8+ • Python 3.10+
Production : o2switch / Apache PHP
Base locale : CSV_DB 6 • Table principale : data_paris_I


================================================================================
ARCHITECTURE
================================================================================

    _Base/              → Scripts Python : import des données DVF
    api/                → API PHP : JSON pour le front-end
    backup.py           → Sauvegarde locale du site en archive ZIP horodatée
    config.php          → Connexion MySQL partagée (get_pdo())
    config.local.php    → Connexion locale XAMPP (non versionné)
    config.o2switch.php → Connexion production o2switch (non versionné)
    lib/cache.php       → Cache fichier serveur pour les API
    .cache/api/         → Fichiers de cache générés automatiquement
    index.php           → Page d'accueil : recherche d'adresse
    estimation.php      → Page outil : filtres + lancement estimation
    results.php         → Page résultats : comparables + PDF
    lib/db.php          → Alias DB léger vers config.php
    .htaccess           → DirectoryIndex, redirections .html, sécurité, cache assets
    docs/NOTICE.txt     → Documentation technique complète
    TACHES.txt          → Suivi des améliorations en cours


================================================================================
1. PIPELINE D'IMPORT DES DONNÉES
================================================================================

DESCRIPTION
-----------
Importe les fichiers CSV Valeurs Foncières dans la base MySQL en 4 étapes :
filtre commune → dédoublonnage → validation → import.

LANCEMENT (script maître tout-en-un — recommandé)
--------------------------------------------------
Lance toutes les étapes dans l'ordre sur tous les fichiers annuels :

    cd /Applications/XAMPP/xamppfiles/htdocs/estimatiz/_Base
    source .venv/bin/activate
    python3 _run_pipeline_complet.py

Ce script :
    0. Vide la table MySQL (TRUNCATE data_paris_I)
    1. Supprime tous les anciens fichiers *_75.csv
    2. Refiltre chaque ValeursFoncieres-20XX.csv → ValeursFoncieres-20XX_75.csv
    3. Applique dédup + validation + import sur chaque fichier filtré

À relancer à chaque ajout d'un nouveau fichier annuel ou modification des règles.

ÉTAPE 0 (maître) — Filtre par commune
--------------------------------------
Lit chaque ValeursFoncieres-20XX.csv (fichier national) et extrait uniquement
les lignes dont le Code postal est dans 75001–75020.
Produit : ValeursFoncieres-20XX_75.csv

Script individuel : _FIltre_data_par_commune.py


ÉTAPE 1 — Dédoublonnage
------------------------
Le format DVF génère plusieurs lignes par vente quand plusieurs locaux sont
vendus ensemble (appartement + cave, parking...). Le script détecte ces
doublons en comparant les colonnes clés :

    Date mutation · Valeur fonciere · No voie · B/T/Q · Type de voie
    Code voie · Voie · Code postal · Commune · 1er lot

PRIORITÉ en cas de doublon (ordre décroissant) :
    1. Type local   : Appartement (10) > Maison (9) > autres (3) > Dépendance (1)
    2. Surface réelle bâtie : à rang égal, on préfère la ligne qui l'a renseignée

Après dédup, les lignes sans aucune surface (ni Surface reelle bati,
ni aucun des 5 lots Carrez) sont supprimées.

Script individuel : "1_nettoye les doublons avant de preparer le fichier.py"


ÉTAPE 2 — Validation & normalisation
--------------------------------------
Valide chaque colonne selon les règles de Table SQL.txt.
Normalise avant validation : No voie (9001.0→9001), Code type local (1.0→1),
Nombre pieces principales (3.0→3), surfaces (100.0→100).
Si des erreurs sont trouvées → rapport dans _validation_erreurs_csv.txt
et import annulé pour ce fichier.

Script individuel : 2_Validation_&_Normalise_csv.py


ÉTAPE 3 — Import MySQL
-----------------------
Insert par lots de 2000 lignes. Normalise à nouveau lors de l'insertion.
Les valeurs vides sont converties en NULL.

Script individuel : 3_import_csv_to_mysql.py


PARAMÈTRES MODIFIABLES (en haut de _run_pipeline_complet.py et 0_pipeline_complet.py)
--------------------------------------------------------------------------------------
    DB_HOST/PORT  → connexion MySQL   (défaut : 127.0.0.1:3306)
    DB_USER       → utilisateur       (défaut : root)
    DB_PASSWORD   → mot de passe      (défaut : vide)
    DB_NAME       → base de données   (défaut : CSV_DB 6)
    TABLE_NAME    → table cible       (défaut : data_paris_I)
    BATCH_SIZE    → lignes par lot    (défaut : 2000)
    CODES_PARIS   → codes postaux     (défaut : 75001–75020)

    XAMPP démarré avec MySQL actif
    Colonne Id_Data en AUTO_INCREMENT dans data_paris_I

    api/prix-m2.php     → cache serveur 6 heures
    api/autocomplete.php → cache serveur 5 minutes

--------------------------------------------------------------------------------
api/autocomplete.php
--------------------------------------------------------------------------------
RÔLE : Autocomplétion d'adresse en temps réel.

APPEL : GET ?q=<saisie> [&limit=20] [&commune=<ville>]

CACHE :
    Cache serveur 5 minutes.
    Clé : q + commune + limit.
    Dossier : .cache/api/

PARSING AUTOMATIQUE DE LA SAISIE :
    "12 bis avenue de la République, Paris 11"
     → no_voie=12, btq=bis, type=avenue, voie="de la republique"
     → commune=PARIS 11, cp=75011

    Virgule = séparateur adresse / ville.
    Sans virgule : détection automatique de la ville en fin de saisie.
    Paris/Lyon/Marseille + arrondissements reconnus (chiffres ou romains).
    Plus de 30 abréviations de types de voie reconnues (bd, av, imp, sq...).

NORMALISATION DE LA VOIE (dictionnaire abréviations) :
    Les formes longues saisies par l'utilisateur sont converties mot par mot
    vers les abréviations utilisées dans DVF, avant la requête SQL :
        SAINT    → ST       SAINTE   → STE
        FAUBOURG → FBG      FG       → FBG
        GENERAL  → GAL      MARECHAL → MAL
        DOCTEUR  → DR       PRESIDENT→ PDT
    Exemples :
        "saint antoine"         → recherche "ST ANTOINE"
        "faubourg saint martin" → recherche "FBG ST MARTIN"
        "general appert"        → recherche "GAL APPERT"
    Les formes déjà abrégées (st, fbg, gal…) passent inchangées.
    Note : FBG et FG ont été retirés du dictionnaire des types de voie
    pour ne pas être consommés comme type et rester dans les tokens voie.

STRATÉGIE DE RECHERCHE (5 paliers) :
    S  → voie prefix% + commune détectée en fin de saisie
    A  → voie prefix% + commune/CP explicites
    B  → voie prefix% seule
    B2 → voie %LIKE% (fallback si début de voie inconnu)
    C  → commune/CP seuls

RÉPONSE :
    [{ label, code_voie, commune, cp, type_voie, voie, no_voie, btq, section }]
    Données retournées directement depuis la DB (format DVF, abréviations incluses).
    Si aucun résultat : retourne la saisie brute (permet une saisie libre).

api/surface.php
--------------------------------------------------------------------------------
RÔLE : Surface indicative d'un bien à partir des ventes existantes.
       Priorité : Surface Carrez du 1er lot, sinon Surface réelle bâtie.

APPEL : GET ?code_voie=<cv> &commune=<com>
             [&no_voie=] [&btq=] [&type_voie=] [&voie=] [&pieces=]

NIVEAUX DE RECHERCHE :
    exact  → adresse complète (no_voie + type_voie + voie + commune)
    street → fallback par code_voie + commune

RÉPONSE :
    { ok, used_level, count, surface (médiane arrondie),
      stats: { p25, median, p75 }, samples, unit }


--------------------------------------------------------------------------------
api/mutations.php
--------------------------------------------------------------------------------
RÔLE : Liste des ventes (mutations) pour une adresse ou une rue.
       Utilisé pour afficher les comparables dans results.php.

APPEL : GET ?code_voie=<cv> &commune=<com>
             [&no_voie=] [&type_voie=] [&voie=]
             [&surface_min=] [&surface_max=] [&pieces=]

NIVEAUX DE RECHERCHE :
    exact             → no_voie + voie + commune (+ type_voie si fourni)
    street_by_code    → code_voie + commune
    street_by_name    → voie exact + commune (+ type_voie si fourni)

COMMUNE : Match tolérant — Paris/Lyon/Marseille reconnus avec arrondissements
          (paris% · lyon% · marseille%)

FILTRE SURFACE : surface_min et surface_max appliqués sur Carrez sinon Réelle.
                 Les lignes sans surface sont toujours conservées.

FILTRE PIÈCES : pieces (entier) → AND `Nombre pieces principales` = :pieces
                Si absent ou vide, aucun filtre pièces n'est appliqué.

RÉPONSE :
    { ok, used_level, count, rows: [...], stats: { count, mean, median,
      p25, p75, min, max, date_min, date_max } }

    Chaque ligne : { adresse, code_postal, valeur_fonciere, surface,
                     prix_m2, nb_pieces, date_mutation, lots_array }


--------------------------------------------------------------------------------
api/estimate.php
--------------------------------------------------------------------------------
RÔLE : Calcule une estimation de prix pour un bien.

APPEL : GET ?code_voie=<cv> &commune=<com> &section=<sec>
             [&type_local=Appartement] [&pieces=] [&surface=]
             [&annee_min=2010] [&annee_max=2099] [&nmin=10]

ALGORITHME :
    1. Recherche des comparables par scopes successifs (s'arrête dès nmin atteint) :
         street (pièces exactes) → street (±1 pièce) → section → commune → fallback
    2. Filtre IQR 1.5× sur le prix/m² (si assez de comparables)
    3. Calcul des percentiles p20 / médiane / p80
    4. Estimation = percentile × surface du bien

SCOPES :
    street              → même rue (code_voie)
    street_pieces_tol.  → même rue, ±1 pièce
    section             → même section cadastrale
    commune             → toute la commune
    fallback            → le scope avec le plus de résultats

COMMUNE : Même logique LIKE que mutations.php.

RÉPONSE :
    { ok, scope_used, filters_applied, counts: { raw, after_iqr },
      price_sqm: { p20, median, p80 },
      estimate: { low, mid, high, confidence } }

    confidence : score 0→1 basé sur le nombre de comparables et la dispersion.

================================================================================
3. INTERFACE — ARCHITECTURE 3 PAGES
================================================================================

PAGES :
    index.php       → Page d'accueil : recherche d'adresse uniquement
    estimation.php  → Page outil     : filtres + lancement de l'estimation
    results.php     → Page résultats : comparables + estimation + PDF

FLUX UTILISATEUR :

    1. index.php — Recherche d'adresse
       → L'utilisateur tape une adresse
       → autocomplete.php retourne des suggestions (délai 250ms, cache JS session)
       → Il sélectionne une suggestion → bouton "Continuer →" s'active
       → Clic "Continuer →" → navigation vers estimation.php avec l'adresse en URL

       Transfert index → estimation : paramètres URL
           estimation.php?label=...&code_voie=...&commune=...&cp=...
                          &type_voie=...&voie=...&no_voie=...&btq=...&section=...
       Avantage : URL bookmarkable, pas de sessionStorage à ce stade.

    2. estimation.php — Configuration de l'estimation
       → Lit les paramètres URL → affiche l'adresse dans un bandeau bleu
       → Appelle surface.php au chargement → remplit Surface, Surface min (p25), Surface max (p75)
       → L'utilisateur ajuste si besoin :
           • Bloc "Votre bien"               → Surface du logement à estimer
           • Bloc "Filtres des comparables"  → Surface min/max · Nombre de pièces (1–6+)
       → "Réinitialiser les filtres" vide les champs des deux blocs
       → "Visualiser les résultats" → appelle mutations.php
          → stocke les données dans sessionStorage → navigue vers results.php

       Transfert estimation → results : sessionStorage (clé : estimatiz_results)
           Contenu : label, surface, surfaceMin, surfaceMax, pieces,
                     mutData, suggestion (objet adresse complet)

    3. results.php — Résultats et PDF
       → Lit le sessionStorage → affiche les comparables dans un tableau
       → Colonnes : Adresse · Valeur foncière · Surface · €/m² · Date · Pièces · Lot(s)
       → Estimation dynamique recalculée à chaque sélection/déselection (P20/médiane/P80)
       → Indice de confiance basé sur dispersion et nombre de comparables (IQR 1.5×)
       → "← Modifier l'estimation" → reconstruction de l'URL estimation.php?params
         depuis la clé suggestion du sessionStorage → retour avec formulaire restauré
       → "Générer mon PDF" → window.print()

RETOUR results → estimation :
    Le bouton "← Modifier l'estimation" reconstruit dynamiquement l'URL
    estimation.php?params depuis l'objet suggestion stocké dans sessionStorage.
    Au chargement d'estimation.php, si le sessionStorage contient des données
    pour la même adresse (même code_voie + commune), le formulaire est restauré
    (surface, min, max, pièces) sans rappeler surface.php.
    Si l'adresse est différente, surface.php est rappelé normalement.

GÉNÉRATION PDF :
    window.print() déclenche l'impression native du navigateur.
    Le @media print de results.php :
        - Affiche un entête print (adresse, filtres, date) + pied de page (source DVF)
        - Cache les boutons, contrôles et barre de stats
        - Masque les lignes non cochées (tbody tr:not(.selected))
        - Masque la colonne checkbox
        - Format A4 (14mm de marges) via @page
    Sur mobile : "Imprimer → Enregistrer en PDF" (iOS/Android Chrome natif).
    Aucune bibliothèque externe, aucun popup.

RESPONSIVE MOBILE :
    index.php      : inputs pleine largeur, logo et titres réduits sous 600px.
    estimation.php : topnav sticky, blocs en colonne, inputs pleine largeur sous 600px.
    results.php    : tableau avec scroll horizontal (overflow-x:auto), header en
                      colonne, bouton PDF pleine largeur, colonnes estimation empilées.

TOPNAV estimation.php :
    Logo complet affiché : icône SVG Estimatiz + nom "Estimatiz" + tagline
    "Précision • Transparence • Data". Clic sur le logo → retour index.php.
    Sur mobile (< 600px) : tagline masqué, icône réduite à 32px, hauteur 56px.

AIDE À LA SAISIE (index.php) :
    Deux encarts informatifs sous la barre de recherche :
        📍 "Adresse précise" : numéro + rue + ville → ventes au plus proche du bien
        🏘️ "Rue entière"     : rue + ville sans numéro → échantillon plus large
    Placeholder mis à jour pour montrer les deux formes possibles.
    Sur mobile : les deux encarts s'empilent verticalement.

NOTES TECHNIQUES :
    - L'estimation dans results.php est recalculée en JS (filtre IQR 1.5×, P20/P50/P80).
    - surface.php, mutations.php et estimate.php supportent le paramètre &pieces=.
    - mutations.php supporte aussi &surface_min= et &surface_max=.
    - index.php ne contient plus aucun appel API ni logique de formulaire.
    - Les anciennes URLs .html sont redirigées vers les pages .php par .htaccess.
    - Les API prix-m2.php et autocomplete.php utilisent un cache fichier serveur.

5. SAUVEGARDE LOCALE
================================================================================

SCRIPT
------
    Sauvegarde : backup.py
    Restauration : restore.py

    Rôle de backup.py : créer une archive ZIP horodatée du site local.
    Rôle de restore.py : restaurer une archive ZIP existante vers le site local.

LANCEMENT
---------
    Depuis la racine du projet :

        cd /Applications/XAMPP/xamppfiles/htdocs/estimatiz
        python3 backup.py

DESTINATION
-----------
    Les sauvegardes sont créées dans :

        ~/Documents/Sauvegarde Estimatiz/

    Nom du fichier généré :

        estimatiz_backup_YYYYMMDD_HHMMSS.zip

CONTENU ET EXCLUSIONS
---------------------
    Inclus :
        - code PHP, CSS, JS, documentation et scripts utiles
        - fichiers de configuration présents localement

    Exclus automatiquement :
        - dossiers .git, .venv, __pycache__, .cache
        - fichiers CSV du dossier _Base/

    Objectif : conserver une sauvegarde légère du site et de sa configuration,
    sans embarquer les gros fichiers DVF ni les caches générés.

RESTAURATION
------------
    1. Ouvrir restore.py
    2. Remplacer la valeur de BACKUP_ZIP par le fichier ZIP à restaurer :

        BACKUP_ZIP = Path.home() / "Documents" / "Sauvegarde Estimatiz" / "estimatiz_backup_YYYYMMDD_HHMMSS.zip"

    3. Lancer depuis la racine du projet :

        python3 restore.py

    4. Confirmer en tapant :

        EFFACER ET RESTAURER

    Comportement :
        - supprime d'abord les fichiers et dossiers actuels du site
        - conserve toujours le dossier _Base/
        - dans _Base/, supprime tout sauf les fichiers CSV
        - restaure ensuite les fichiers et dossiers du ZIP, y compris les fichiers
          cachés comme .htaccess ou .gitignore s'ils sont présents dans l'archive
        - crée automatiquement les dossiers manquants
        - n'efface jamais et n'écrase jamais les fichiers CSV du dossier _Base/
        - refuse de démarrer si le ZIP à restaurer est placé dans le dossier du site
